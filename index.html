Understanding Threads & Multithreading

🧵 What is a Thread?

  • A thread is the smallest unit of execution within a process.
  • A single program can have multiple threads running concurrently.
  • All threads in a process share the same memory (heap) but have their own stack.
  • Java has built-in multithreading support via java.lang.Thread.
🌍 Real World Analogy

Think of a restaurant kitchen. The chef (CPU) can cook multiple dishes (threads) simultaneously. Each dish is being prepared independently but uses the same kitchen (process memory).

⚡ Need for Multithreaded Programming

  • Better CPU utilization — don't waste CPU while waiting for I/O.
  • Responsiveness — UI thread stays alive while background work happens.
  • Parallelism — perform multiple computations simultaneously on multi-core CPUs.
  • Resource sharing — threads share memory without IPC overhead.
  • Scalability — handle thousands of client requests in servers.

🏗️ Creating Threads — 2 Ways

  • Method 1: Extend Thread class and override run().
  • Method 2: Implement Runnable interface and pass to Thread.
  • Method 3 (modern): Use Callable + ExecutorService for result-returning tasks.
  • ✅ Prefer Runnable — Java has single inheritance, so extending Thread blocks you from extending other classes.

📋 States Explained

  • NEW: Thread object created, start() not called yet.
  • RUNNABLE: Ready to run, waiting for CPU time.
  • RUNNING: Currently executing on CPU.
  • BLOCKED: Waiting to acquire a lock (synchronized block).
  • WAITING: Indefinitely waiting (wait(), join()).
  • TIMED_WAITING: Waiting for specified time (sleep(ms), wait(ms)).
  • TERMINATED: Thread has finished execution.

Thread Priorities, Synchronization & Deadlock

🎚️ Thread Priorities

  • Priority range: 1 (MIN_PRIORITY) to 10 (MAX_PRIORITY), default is 5 (NORM_PRIORITY).
  • Set with thread.setPriority(n), get with thread.getPriority().
  • Higher priority threads get CPU time first — but not guaranteed (OS-dependent).
  • Don't rely solely on priorities for correctness — use synchronization.

🔒 Synchronization

  • Prevents race conditions when multiple threads access shared data.
  • Uses the intrinsic lock (monitor) of an object.
  • Synchronized method: public synchronized void method() — locks the whole object.
  • Synchronized block: synchronized(obj){ ... } — finer control, better performance.
  • Only one thread can execute synchronized code on the same object at a time.

💬 Inter-Thread Communication

  • Methods from Object class: wait(), notify(), notifyAll().
  • wait() — releases lock and pauses thread until notified.
  • notify() — wakes one waiting thread.
  • notifyAll() — wakes all waiting threads.
  • Must be called inside a synchronized block, else IllegalMonitorStateException.
  • Classic use case: Producer-Consumer problem.

💀 Deadlock

  • Deadlock = two or more threads indefinitely waiting for each other to release locks.
  • 4 Conditions (Coffman's): Mutual Exclusion, Hold & Wait, No Preemption, Circular Wait.
  • Prevention: Always acquire locks in the same order, use tryLock() with timeouts.
  • Detection: JVM thread dump shows BLOCKED threads in circular dependency.
🌍 Real World Analogy

Person A holds a pen and wants paper. Person B holds paper and wants a pen. Both wait forever — DEADLOCK!

⚙️ Java Concurrency Framework (java.util.concurrent)

  • ExecutorService — manages a pool of threads, no manual thread creation.
  • ThreadPoolExecutor — configurable thread pool.
  • Callable<T> — like Runnable but returns a value and can throw exceptions.
  • Future<T> — holds the result of async computation, use future.get().
  • Lock interface — ReentrantLock more flexible than synchronized.
  • Semaphore — controls access to a limited number of resources.
  • CountDownLatch — wait for multiple threads to finish before proceeding.
  • CyclicBarrier — like latch but reusable.

Java IO Package & Streams

📦 java.io Package

  • Core package for all standard I/O operations in Java.
  • Two types: Byte Streams (InputStream/OutputStream) and Character Streams (Reader/Writer).
  • Character streams handle Unicode automatically (use for text files).
  • Byte streams handle raw binary data (images, audio, etc.).

🌊 Stream Hierarchy

  • InputStream → FileInputStream, BufferedInputStream, DataInputStream, ObjectInputStream
  • OutputStream → FileOutputStream, BufferedOutputStream, DataOutputStream, ObjectOutputStream
  • Reader → FileReader, BufferedReader, InputStreamReader
  • Writer → FileWriter, BufferedWriter, OutputStreamWriter, PrintWriter
🌍 Analogy

A stream is like a water pipe. Data flows through it in one direction. You can read from or write to it — not both at the same time.

🚀 Java NIO (New I/O) — java.nio

  • Introduced in Java 1.4 for high-performance, scalable I/O.
  • Key concepts: Buffer, Channel, Selector.
  • Non-blocking I/O — a thread doesn't block waiting for data.
  • Buffer-oriented — data written/read to/from buffers, not directly to streams.
  • Channel — bidirectional (can read AND write), vs stream which is unidirectional.
  • Selector — single thread manages multiple channels (Multiplexing).

📄 File Object & File I/O

  • java.io.File represents a file/directory path (doesn't read/write itself).
  • Useful methods: exists(), createNewFile(), delete(), mkdir(), listFiles(), getName(), getPath().
  • To write: use FileWriter or BufferedWriter.
  • To read: use FileReader or BufferedReader (readLine() is very useful).
  • Always close streams in finally or use try-with-resources.

⚡ Buffers & Buffer Management (NIO)

  • A Buffer is a memory block that temporarily stores data.
  • Types: ByteBuffer, CharBuffer, IntBuffer, LongBuffer, etc.
  • Key buffer properties: capacity, position, limit, mark.
  • flip() — switch from write mode to read mode.
  • clear() — reset buffer for writing again.
  • rewind() — re-read from start without clearing data.

🔗 File Channels

  • FileChannel — NIO channel for reading/writing files.
  • Works with ByteBuffer — read/write data in bulk.
  • Supports memory-mapped files (map()) for extremely fast I/O.
  • Supports file locking (lock(), tryLock()).
  • Get it via: FileChannel.open(path) or FileInputStream.getChannel().

💾 Serialization

  • Serialization = converting an object into a byte stream for storage/transmission.
  • Deserialization = reconstructing the object from the byte stream.
  • Implement java.io.Serializable interface (marker interface — no methods).
  • Use ObjectOutputStream.writeObject(obj) to serialize.
  • Use ObjectInputStream.readObject() to deserialize.
  • transient keyword — field skipped during serialization.
  • serialVersionUID — version control for serialized classes.
🌍 Analogy

Serialization is like converting a 3D sculpture to a blueprint. Deserialization is rebuilding the sculpture from the blueprint.

👁️ Observer & Observable (Design Pattern)

  • Observer Pattern = defines one-to-many dependency. When subject changes, all observers are notified.
  • java.util.Observable (deprecated in Java 9) — the subject class.
  • java.util.Observer (deprecated) — the interface observers implement.
  • Modern alternative: PropertyChangeListener, RxJava, or event listeners.
  • setChanged() + notifyObservers() — called in Observable to notify all observers.
  • update(Observable o, Object arg) — called on each Observer.

Collection Framework Overview

🏗️ What is the Collections Framework?

  • A unified architecture for representing and manipulating groups of objects.
  • Package: java.util
  • Provides ready-to-use data structures — no need to build from scratch.
  • Key benefits: Reduces programming effort, increases performance, promotes reusability.
🌍 Analogy

The Collections Framework is like a toolbox. Instead of building your own hammer, you just pick the right tool — ArrayList for dynamic lists, HashMap for key-value lookups, etc.

🗺️ Collection Hierarchy (Key Interfaces)

  • Iterable → Collection → List, Set, Queue
  • List — ordered, allows duplicates → ArrayList, LinkedList, Vector, Stack
  • Set — no duplicates → HashSet, LinkedHashSet, TreeSet
  • Queue — FIFO ordering → LinkedList, PriorityQueue, ArrayDeque
  • Map (separate root) — key-value pairs → HashMap, TreeMap, LinkedHashMap

📋 ArrayList vs Vector vs Stack

  • ArrayList: Dynamic array. NOT thread-safe. Fast random access. Default choice for lists.
  • Vector: Like ArrayList but synchronized (thread-safe). Slower. Legacy class.
  • Stack: Extends Vector. LIFO (Last In First Out). Methods: push(), pop(), peek(), isEmpty().
  • LinkedList: Doubly-linked list. Efficient insert/delete. Implements both List and Deque.
  • Queue (LinkedList): FIFO. Methods: offer(), poll(), peek().

🎯 Set Interface

  • No duplicate elements allowed.
  • HashSet: Backed by HashMap. Fastest. No order guaranteed.
  • LinkedHashSet: Maintains insertion order. Slightly slower than HashSet.
  • TreeSet: Sorted order (natural or custom Comparator). Backed by TreeMap. O(log n) operations.
  • Use equals() and hashCode() for duplicate detection.

⚖️ Key Comparisons at a Glance

  • ArrayList vs LinkedList: ArrayList better for search/access (O(1)), LinkedList better for insert/delete (O(1)).
  • HashSet vs TreeSet: HashSet O(1) but unordered, TreeSet O(log n) but sorted.
  • ArrayList vs Vector: ArrayList faster (not synchronized), Vector thread-safe but slow.
  • Stack vs ArrayDeque: ArrayDeque preferred over Stack (faster, no legacy overhead).
